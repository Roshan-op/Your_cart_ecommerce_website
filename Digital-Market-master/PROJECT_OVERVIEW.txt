1. Executive summary
--------------------
Digital Market is a full-stack, demo e-commerce application composed of a Django REST backend and a Create-React-App (CRA) frontend. The repository contains product data (CSV), an item-to-item recommendation component driven by a precomputed similarity matrix, user authentication (JWT), product and order management endpoints, and a React + Redux storefront with cart and checkout flows.

This document summarizes architecture, data, code locations, the current recommendation algorithm in use, how data is ingested, and concrete options for improving recommendation quality, scaling, and evaluation — suitable for inclusion in a project proposal.

6. Recommendation implementation (current)
------------------------------------------
What the repo contains:
- A precomputed similarity matrix file (`similarity9.pkl`) is loaded at module import in `backend/base/views/product_views.py`:

  similarity = pickle.load(open('similarity9.pkl', 'rb'))

- The `recommend` view function (GET `/api/products/<pk>/recommend/`) does the following:
  1. Load the Product by primary key `_id=pk`.
  2. Map product `_id` to a zero-based index via `idx = product._id - 1` (this assumes the `similarity` matrix rows/columns align with product dataset order and that `_id` increments starting at 1 in that same order).
  3. Fetch the `distances` row: `distances = similarity[idx]` (an array-like row of similarity scores between this product and all products).
  4. Sort enumerated distances descending, skip the first element (likely the product itself at highest similarity), and choose the next top-k indices: `product_indices = sorted(list(enumerate(distances)), reverse=True, key=lambda x: x[1])[1:5]` (returns 4 recommended indices).
  5. Map each index back to product `_id` with `pid = i + 1` and fetch the corresponding Product objects to return serialized results.

Important notes and constraints about the current approach:
- `similarity9.pkl` is not generated by any script currently present in repo (no explicit code found that creates it). The system expects the matrix to be aligned to product ordering in the dataset. If dataset order or DB `_id` assignment changes, the mapping will break.
- The algorithm uses a precomputed item-item similarity matrix and performs an O(n) lookup per request (fetch row + sort top-k). This is inexpensive at query time but requires an O(n^2) precomputation and O(n^2) storage for the matrix.
- The code uses raw similarity values (bigger is more similar) and picks top scores; there is no thresholding, diversification, or business-rule filtering (e.g., stock availability, vendor restrictions) in the provided view.

7. How `similarity9.pkl` could have been built (recommended reproduction)
-------------------------------------------------------------------------
There is no creation script in the repo. A common, practical way to produce such a matrix for content-based recommendations is:

- Build a product text field to vectorize, e.g. combine fields: `names + brand + category + description`.
- Vectorize the text using TF-IDF or another text-vectorizer: `TfidfVectorizer(stop_words='english')`.
- Compute cosine similarity between vectors to get an NxN matrix.

Example (python, requires scikit-learn & pandas):

```python
import pandas as pd
import pickle
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.metrics.pairwise import cosine_similarity

# load original CSV (ensure ordering matches DB import order)
df = pd.read_csv('data.csv')
# assemble text
text = (
    df['names'].fillna('') + ' ' + df['brand'].fillna('') + ' ' + df['category'].fillna('') + ' ' + df['description'].fillna('')
)

vectorizer = TfidfVectorizer(max_features=20000, stop_words='english')
X = vectorizer.fit_transform(text)

# cosine similarity (dense NxN may be large for big catalogs)
sim = cosine_similarity(X)

# save
with open('similarity9.pkl', 'wb') as f:
    pickle.dump(sim, f)
```

Mathematics: cosine similarity between two TF-IDF vectors $\mathbf{u}$ and $\mathbf{v}$ is computed as:

$\displaystyle \text{sim}(\mathbf{u},\mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\|\,\|\mathbf{v}\|}$

Notes / caveats:
- For large catalogs ($n$ in tens/hundreds of thousands), storing full dense $n\times n$ matrices is infeasible (memory and compute). Use Approximate Nearest Neighbors (ANN) libraries (FAISS, Annoy) or compute top-k neighbors during precomputation and store only sparse neighbor lists.
- Keep an explicit mapping between dataset row index and `Product._id` to avoid brittle assumptions. Example: create and save `index_to_product_id.json` when generating similarity matrix, then use that mapping in `recommend()`.

8. Alternative / improved recommendation approaches (detailed)
-------------------------------------------------------------
Below are options ordered by increasing complexity and quality.

A) Tuned content-based TF-IDF + cosine (current approach, explicit mapping)
- Vectorize combined product text (TF-IDF) or categorical features (one-hot / embedding) and compute cosine similarity.
- Pros: simple, interpretable, no user history required.
- Cons: cold-start for new products, misses collaborative signals.

B) Item-based collaborative filtering
- Build user-item interactions (ratings, purchase events) matrix $R$ and compute item-item similarities using cosine or Pearson correlation on columns of $R$.
- Pros: captures real user behavior.
- Cons: requires sufficient historical interactions.

C) Matrix factorization (ALS or SVD)
- Factorize user-item matrix $R \approx U V^T$. Use products' latent vectors to compute similarity; this supports recommendation for users and items.
- Libraries: implicit (ALS for implicit feedback), Surprise (explicit ratings), LightFM.

D) Embedding-based content representations (modern)
- Use pretrained language models (SentenceTransformers/BERT) to embed product descriptions and compute similarity in embedding space. Optionally fine-tune with product titles.
- Compute ANN index (FAISS) for fast nearest-neighbor search.
- Pros: better semantic matching than TF-IDF.
- Cons: heavier compute; model / GPU needed for indexing training.

E) Hybrid methods
- Combine content-based (text) and collaborative signals (interactions) using weighted scoring or learning-to-rank models.
- Use features such as brand, category, price, availability, vendor to filter or re-rank results.

F) Production concerns
- Precompute top-k neighbors (e.g., top-50 per item) and store as JSON or in DB for fast lookup.
- Use incremental updates: compute embeddings for new products and query ANN to fill neighbor lists rather than recomputing full NxN.
- Cache recommendations per product in Redis with TTL.

9. Evaluation & metrics (how to measure)
----------------------------------------
Offline metrics and approaches:
- Holdout a set of interactions (or simulated co-purchase pairs) and measure top-k metrics:
  - Precision@K, Recall@K
  - Mean Average Precision (MAP)
  - Normalized Discounted Cumulative Gain (NDCG)

Formulas:
- Precision@K for a query set $Q$:
  $\text{Precision@K} = \frac{1}{|Q|} \sum_{q \in Q} \frac{|\text{relevant}_q \cap \text{topK}_q|}{K}$

- NDCG@K:
  $\text{NDCG}@K = \frac{1}{|Q|} \sum_{q\in Q} \frac{\sum_{i=1}^{K} \frac{2^{rel_{q,i}}-1}{\log_2(i+1)}}{\text{IDCG}_q}$

A/B testing:
- Deploy alternate recommender (e.g., baseline TF-IDF vs. new embedding model), randomize users, and measure online metrics such as click-through rate (CTR), add-to-cart rate, and conversion rate.

13. Scalability & deployment recommendations
--------------------------------------------
- Small dataset (hundreds to low thousands of items): precompute and store dense similarity matrix or per-item neighbor lists; lookup in memory or from a local pickle is fine.
- Medium/Large catalog (10k+ items): use embedding + ANN (FAISS) and store only per-item top-k neighbor lists; compute embeddings offline, store to object storage (S3) and load into memory or serve via a microservice.
- Latency optimization: compute neighbors offline (batch), store as compressed neighbor lists (e.g., top-50) in a DB table or Redis; explore caching per-product recommendations.
